## 语料量能塞进上下文窗口，就先别上 RAG
在规划"数字分身"项目时的判断：playbook 加飞书文档总量大概率能塞进一个 200K token 的上下文窗口，这意味着 MVP 阶段根本不需要 RAG 管线——全量语料 + persona prompt 直接喂给模型就是一个能用的分身。RAG 是语料涨到几百篇之后才需要解决的问题。避免了为一个几十篇文档规模的语料库过早搭建检索基础设施。

## 模糊需求词往往盖着好几个完全不同的产品，动手前先分岔
"分身"这个词掩盖了三个完全不同的产品：A. 给自己用的第二大脑（检索问题）；B. 对外的专家分身（Delphi 模式，个人品牌资产化）；C. 内部代理人（新人问答）。三者的技术路径和评价标准完全不同，必须先选一个再决定怎么做，而不是直接开工。结合自己在做 infiDive（命门是 Meaningful Memory Trigger 和用户模型）的背景，判断出真正的"元需求"其实是把"给自己做分身"当作技术路线的 dogfooding——用自己当第一个用户，验证"多大的语料密度+什么样的记忆结构，才能让 AI 答出像这个人的回答"，这个经验直接反哺主产品。

## 语料能撑住"事实检索"不代表能撑住"观点生成"，冷启动要按能力上限降级定位
当前瓶颈是数据不是技术：20 篇文档能支撑"回答关于你方法论的问题"（事实检索），撑不起"模拟你的判断"（观点生成）——后者需要的是高密度的观点语料：具体问题的取舍、反对什么、决策案例。所以冷启动定位应该降级为"playbook 问答入口"，随语料增长再进化成"分身"。这也反过来给写作一个飞轮：每篇手记都在给未来的分身喂料。

## 知识沉淀管线要分"机器候选区"和"人工确认区"，机器只产候选不直接转正
每日沉淀管线设计：增量镜像飞书+聊天 → commit/push → Claude 读当日变更语料、按"宁缺毋滥"标准提炼 → 写入 `ai-product-playbook/inbox/日期.md` → 自动 push。inbox 是机器写入区，正文目录仍归人工整理——和自己 Memory pilot 的"候选→人工确认"习惯保持一致。机器负责产出候选，人工负责把关转正，两者不合并。

## 标注类项目的难度可以类比自动驾驶感知融合来分级，训练策略按"做加法"还是"做减法"设计
经验说明：类比自动驾驶项目的 2D、3D、2/3D 融合、融合多帧；LLM 项目即为判断评分（分类/排序）、改写转写、判断评分+改写转写、多轮标注，难度依次升高。判断评分类任务标注员的困难点在于常识储备、客户规则不够明晰，需要项目经理对培训内容做加法、扩充尝试；改写转写类任务的困难点在于大量术语，需要项目经理对规则内容做减法，得能用几句话讲清楚项目在做什么，此处参考降序法：存在安全问题→重大事实错误（答非所问）→部分事实错误（延伸信息错误）→存在重复语句/语病但无事实错误→条理清晰回答准确但语句表述不优美→"信-雅-达"；判断评分+改写转写类任务既须对前者做加法还须对后者做减法，先让标注员能判断对，再掌握改写转写技能，最后关联；多轮标注难点在于标注员不了解如何进行答题，需要分模块培训规则并考核。总体做题思路：看题型——套模型/步骤——选择/写答案。这套经验说明被固化成模板，复用在了运动类目、3D 载具、局部特效、手写作业、图生图、一致性等多个不同项目的管理文档里。

## 标注员薪酬结算依据只能是验收通过量，不能是产量
项目管理硬性规则：标注员工资结算依据只能是【验收通过数量】而非产量。按产量结算会激励标注员追求数量、牺牲质量；按验收通过量结算才能把激励对齐到最终交付质量。这条规则同样被固化进了多个项目的管理文档模板里，跨项目复用。
