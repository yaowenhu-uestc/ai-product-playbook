# Playbook 更新判断清单

> **第三部 · Prompt · 第31章**
>
> 撰写日期：2026-07-03

## 结论

AI Product Playbook 不需要每天强行更新。更好的节奏是：

> 每天检查一次，但只在出现可复用的产品判断、方法、Prompt、Workflow、模板或资源时更新。

Playbook 的价值不在更新频率，而在筛选质量。空更新是正常结果，不是维护失败。

## 什么时候值得更新

| 信号 | 是否更新 | 适合放入 |
| --- | --- | --- |
| 出现新的产品判断 | 更新 | `product-notes/` |
| 形成可复用 Prompt | 更新 | `prompts/` |
| 发现新的 Agent / Workflow 方法 | 更新 | `agent/` |
| 总结出稳定操作模板 | 更新 | `learning/` 或 `skills/` |
| 收集到高价值工具、文章、论文 | 更新 | `resources/` |
| 只是一次性闲聊或情绪表达 | 不更新 | 留在聊天 |
| 只是执行了普通任务 | 不更新 | 留在任务记录 |
| 涉及隐私或内部细节且无法抽象 | 不更新 | 不进入公开仓库 |

## 三层筛选

更新前先把内容分成三层：

| 层级 | 说明 | 处理 |
| --- | --- | --- |
| Chat Noise | 临时表达、重复讨论、一次性上下文 | 丢弃 |
| Candidate Note | 有价值但还不稳定的想法 | 先记录为草稿或短模板 |
| Playbook Entry | 可复用、可公开、能影响后续判断 | 写入仓库并更新导航 |

判断一个内容是否进入 Playbook，可以问四个问题：

1. 三个月后我还会需要这个判断吗？
2. 另一个 AI 产品经理能不能复用它？
3. 它是否能被写成方法、模板、Prompt、Workflow 或资源？
4. 去掉私人经历和内部细节后，它是否仍然成立？

如果四个问题大多为“否”，就不要更新。

## 更新边界

| 可以写 | 不写 |
| --- | --- |
| 我对 AI 产品、Workflow、Memory、Agent 的判断 | 原始聊天记录 |
| 抽象后的产品方法和工作流 | 公司内部细节 |
| 公开可复用的 Prompt 和模板 | 真实人名、薪资、融资、简历隐私 |
| 经过验证的工具使用经验 | 未验证的宏大判断 |
| 失败经验和取舍逻辑 | 只对当天有效的任务进度 |

公开仓库要保留“我”和“我的判断”，但不把私人经历作为主要内容。核心身份始终是 AI 产品经理 / AI 产品实践者。

## 每日检查 Prompt

```text
请判断本聊天是否值得更新到 AI Product Playbook。

Playbook 是一个公开 GitHub 仓库，用来沉淀我的 AI 产品经理实践，包括产品手记、方法论、Prompt、Skill、Agent、Workflow、资源等。

请先筛选，不要为了更新而更新。只有当本聊天里出现新的产品判断、可复用方法、Prompt、Workflow、模板或资源时，才建议写入。

输出：
1. 是否建议更新：是 / 否
2. 值得沉淀的内容：用表格列出主题、原因、建议目录
3. 不建议沉淀的内容：说明为什么不写
4. 如果建议更新，请给出文件名、标题和 5 条以内的大纲

写作规则：
- 中文为主，公开、克制、专业。
- 可以写“我”和“我的判断”，但少写私人经历。
- 主线身份是 AI 产品经理 / AI 产品实践者。
- 可保留 User Model、Context、Agent、Workflow、Retention、PMF 等术语。
- 涉及具体公司或内部项目时全部抽象，不写内部细节、真实人名、薪资、融资、简历隐私。
- 不原样搬运聊天，要提炼成产品手记、方法论、模板、Prompt 或资源。
```

## 我的判断

Playbook 应该像产品记忆系统一样工作：它的质量由“拒绝写入什么”决定。

每天更新不是目标，每天筛选才是目标。真正值得进入仓库的内容，应该能在未来继续影响判断、协作或执行。
