# 从 Prompt 到 Workflow

> **第三部 · Prompt · 第29章**
>
> 撰写日期：2026-07-02

## 结论

Prompt 不是一句“咒语”，而是一个最小工作流。

当一个 Prompt 被反复使用，它就不应该停留在临时文本里，而应该升级为：

| 阶段 | 形态 |
| --- | --- |
| 一次性提问 | Prompt |
| 稳定复用 | Prompt 模板 |
| 多步骤任务 | Workflow |
| 需要工具和文件 | Skill |
| 需要持续执行和状态 | Agent |

## Prompt 的基本结构

```text
角色：
任务：
背景：
输入：
输出格式：
约束：
判断标准：
```

缺少“判断标准”的 Prompt 很难稳定。模型可能给出看似合理的答案，但用户无法判断是否达标。

## 产品分析 Prompt

```text
你是 AI 产品经理。

请分析【产品/功能/场景】，输出中文表格和结论。

背景：
- 目标用户：
- 使用场景：
- 当前问题：

请按以下结构输出：
1. 一句话结论。
2. 用户需求拆解：表格，包含用户、任务、痛点、替代方案。
3. 产品链路拆解：从触发、输入、处理、输出、反馈、留存分析。
4. AI 能力判断：哪些环节适合 AI，哪些不适合。
5. 风险与约束：数据、成本、合规、体验、评估。
6. 我的产品判断：是否值得做，先做什么，为什么。

要求：
- 不写空泛套话。
- 有不确定信息要标注假设。
- 比较和枚举优先用表格。
```

## PRD 辅助 Prompt

```text
你是资深产品经理，请帮我把以下需求整理成 PRD 草稿。

输入信息：
【粘贴背景、用户问题、目标、已有约束】

输出结构：
1. 背景与目标
2. 用户与场景
3. 需求范围
4. 核心流程
5. 功能说明
6. 异常状态
7. 数据指标
8. 验收标准
9. 暂不做事项

要求：
- 对缺失信息列出“待确认问题”。
- 不要替我编造业务事实。
- 每个功能点都要有验收标准。
```

## Agent 评估 Prompt

```text
你是 Agent 产品评估专家。

请评估以下 Agent Workflow 是否适合进入真实使用。

输入：
- 用户目标：
- Workflow 步骤：
- 可用工具：
- 权限边界：
- 成功标准：

请按表格评估：
1. 目标清晰度
2. Context 完整性
3. 工具匹配度
4. 权限风险
5. 失败恢复能力
6. 结果验证方式
7. 用户介入成本

最后给出：
- 最大风险
- 最小可上线版本
- 下一轮测试用例
```

## Prompt 调优流程

| 步骤 | 动作 | 产出 |
| --- | --- | --- |
| 1. 定义标准 | 明确好输出是什么 | 评分规则 |
| 2. 准备样本 | 收集真实输入和理想输出 | 测试集 |
| 3. 写初版 | 先覆盖主路径 | Prompt v1 |
| 4. 批量运行 | 不靠单条样例判断 | 输出集 |
| 5. 分析错例 | 找共性失败原因 | 错误分类 |
| 6. 改写 Prompt | 针对共性问题调整 | Prompt v2 |
| 7. 回归测试 | 防止修一个坏三个 | 对比结果 |

我的判断是，Prompt 调优一旦进入“批量样本 + 指标对比 + 错例复盘”，它就已经不是写提示词，而是在做轻量评测工程。

## 何时升级为 Skill

| 信号 | 说明 |
| --- | --- |
| 同类 Prompt 反复出现 | 需要模板化 |
| 每次都要读同一批资料 | 需要固定 Context |
| 需要调用脚本或 API | 需要工具化 |
| 需要多步骤判断 | 需要 Workflow |
| 需要处理异常 | 需要执行说明 |

Prompt 是起点，不是终点。真正有价值的是把有效 Prompt 变成稳定工作流。
