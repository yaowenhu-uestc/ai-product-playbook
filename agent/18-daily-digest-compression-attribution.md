# 团队每日摘要：压缩纪律与署名纪律

> **第二部 · Agent 实践 · 第27章**
>
> 撰写日期：2026-07-10

## 结论

做团队每日摘要（把一天的群聊、文档变更、会议记录压成一份日报）时，我踩到的两个真问题不在"怎么生成"，而在"怎么克制"和"怎么归属"：内容太多会没人读，署名猜错会伤信任。

> 每日摘要的价值不在覆盖全，而在**能被 20~30 秒扫完**。它是压缩仪式，不是流水账。而它的可信度，取决于每条信息的署名是不是宁可留白也不猜错。

## 1. 压缩纪律：先定硬上限，再往里塞

摘要一旦允许"重要的都放进去"，就会退化成没人看的长列表。有效的做法是先设死物理上限，逼着内容做优先级：

| 约束 | 参考量级 | 作用 |
| --- | --- | --- |
| 总长度 | ≤ 350 字 | 保证 20~30 秒扫完，而不是 5 分钟读 |
| 条目数 | ≤ 8 条 | 逼出"今天最该知道的几件事" |
| 单条长度 | ≤ 40 字 | 只留结论，去掉过程 |

在这些上限下，该被砍掉的是：链接堆、闲聊、过程性动作。该被留下的只有三类——**决策、风险、完成信号**。判断标准很简单：这条不知道，明天会不会做错事？不会，就不进摘要。

## 2. 署名纪律：宁可无名，不可猜名

自动摘要最伤信任的不是漏了一条，而是把一件事安到了错的人头上。所以署名只认可靠来源，不做推断：

| 可靠署名源 | 说明 |
| --- | --- |
| 带发送者的聊天记录 | 消息自带 sender |
| 带作者的文档编辑日志 | 平台元数据里的 editor/creator |
| 带名字的代码提交 | commit author |

绝不做的事：从上下文猜"这话大概是谁说的"、默认会议主持人就是提案人、按角色反推谁负责。

> 默认输出"无名"，好过输出"很可能是某人"。一次猜错的署名，会让整份摘要的可信度打折。

## 3. 内容来源：有些内容要靠"约定"才被发现

不是所有内容都能被系统自动抓到。会议纪要、外部文档这类，往往散落在群聊里的一条链接后面，抓取管道根本看不见正文。与其硬做全网感知，不如**和团队约定一个可被机器识别的投递动作**——例如"会议纪要一律把链接甩进某个群/某个话题"，摘要管道再顺着这个约定去解析链接。

这条经验可以推广：**当自动抓取有盲区时，先建一条低成本的人工约定把内容"喂"到管道能看见的地方，比硬啃感知能力划算得多。**

## 4. 让摘要自己产生反馈信号

日报发出去不该是终点。在卡片里嵌入按钮、做实时计数，就能把"有没有人真的看"变成可度量事件；再叠加一份覆盖率报表（N/M 谁看了），摘要本身就成了一个能自我评估的产品，而不是单向广播。这条与[采纳度量](17-adoption-metrics-small-team.md)是同一套埋点思路。

## 5. 脱敏是对外产出的默认项

摘要一旦可能流出团队边界（进公开归档、喂给外部 Agent），真名就是隐私风险。稳妥做法是双保险：生成阶段用昵称/角色替代真名，导出阶段再过一遍脱敏校验。**脱敏不是发之前补一刀，而是从生成就默认开启。**
